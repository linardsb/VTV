# Fix Error — One-Pass Error Resolution

## INPUT

**Error:** $ARGUMENTS

The input can be:
- A **GitHub issue ID** (e.g., `42`) — fetched via `gh issue view`
- An **error description** with stack trace pasted from GlitchTip
- A **short description** of the error behavior

## PHASE 1: UNDERSTAND THE ERROR (2-3 min)

### 1.1 Gather error details

```
!gh issue view $ARGUMENTS 2>/dev/null || echo "Not a GitHub issue — using provided description"
```

Extract from the input:
- **Error type** (e.g., `RuntimeError`, `ValidationError`, `IntegrityError`)
- **Stack trace** — identify the file, function, and line where it crashes
- **Request context** — URL, method, payload if available
- **Frequency** — one-off or recurring pattern

### 1.2 Trace to source code

Starting from the stack trace (or error description), locate the exact failure point:

1. **Find the crashing function** — Read the file and surrounding context (50 lines above/below)
2. **Trace the call chain** — Follow the route → service → repository path to understand data flow
3. **Check the schema** — If the error involves request/response data, read the Pydantic schema
4. **Check the model** — If the error involves database operations, read the SQLAlchemy model
5. **Check recent changes** — `git log --oneline -10 -- {affected_file}` to see if a recent change caused it

### 1.3 Identify root cause

Determine:
- **What** fails (specific function, line, or interaction)
- **Why** it fails (missing validation, wrong type, race condition, missing field, etc.)
- **Category**: validation gap | logic error | type error | missing constraint | config issue | schema mismatch | import error

## PHASE 2: IMPLEMENT THE FIX (3-5 min)

### 2.1 Apply the code fix

Fix the root cause following VTV conventions:
- Complete type annotations on all new/modified functions
- Structured logging: `domain.component.action_state`
- Async patterns for database operations
- Use `escape_like()` for any ILIKE search params
- Use `DomainValidationError` not `ValidationError` for domain errors

**Known pitfalls (check all that apply):**
- Adding optional fields to schemas? Update ALL constructor call sites (conftest, routes, services, tests)
- Adding validators to input schemas? Will this reject existing production data?
- Changing response fields? Grep `app/*/tests/` and update assertions
- `from datetime import date` in a model with a `date` field? Use `import datetime` + `datetime.date`
- `FastAPI Query(None)`? Add `# noqa: B008`
- New `@field_validator`? Put on the REQUEST schema, never on the shared model

### 2.2 Write regression test

Create a test that would have caught this error:
- **Name**: `test_error_{short_description}()` (e.g., `test_error_calendar_missing_field`)
- **Place** in the affected feature's `tests/` directory
- **Test the fix**: verify the previously-failing scenario now works
- **Test the edge case**: verify related edge cases don't regress
- Use `@pytest.mark.integration` only if the test requires a real database

### 2.3 Run database migrations (if needed)

If the fix requires schema changes:
```bash
uv run alembic revision --autogenerate -m "fix: {short_description}"
```
Review the generated migration — remove any unrelated autogenerated changes.
```bash
uv run alembic upgrade head
```

## PHASE 3: VALIDATE (1-2 min)

Run each check in sequence. Fix any issues before proceeding:

```bash
uv run ruff format .
```

```bash
uv run ruff check --fix .
```

```bash
uv run mypy app/
```

```bash
uv run pyright app/
```

```bash
uv run pytest -x -q --tb=short
```

**Error recovery rules:**
- After ANY code edit to fix a validation error, re-run ALL checks from ruff format
- Maximum 3 fix attempts per check before stopping
- If stuck after 3 attempts, STOP and report the failures to the user
