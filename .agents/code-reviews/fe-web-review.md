### High

**H1. In-memory brute-force protection does not survive restarts**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/auth.ts`, lines 27-51
- **Standard:** Security
- **Details:** The `loginAttempts` Map is stored in-memory. In production with multiple server instances or after a restart, all lockout state is lost. An attacker can bypass the 5-attempt lockout by simply waiting for a redeploy.
- **Recommendation:** Move login attempt tracking to Redis or the database.

**H2. Dashboard page (/) not protected by middleware RBAC**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/middleware.ts`, line 32
- **Standard:** Security
- **Details:** The middleware matcher is:
```typescript
matcher: ["/(lv|en)/(routes|stops|schedules|gtfs|users|chat|documents)/:path*"],
```
This does NOT include the root dashboard path `/lv/` or `/en/`. The dashboard page is accessible without authentication.
- **Recommendation:** Add the dashboard route to the matcher, or use a catch-all authenticated pattern.

**H3. `text-white` and `bg-black` in shadcn/ui base components**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/components/ui/button.tsx`, line 14
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/components/ui/badge.tsx`, line 16
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/components/ui/sheet.tsx`, line 39
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/components/ui/dialog.tsx`, line 42
- **Standard:** Design System Compliance
- **Details:** The shadcn/ui base components use `text-white` on destructive variants and `bg-black/50` for overlay backdrops. While these are generated by shadcn and not application code, they violate the semantic token policy. The correct tokens would be `text-destructive-foreground` and `bg-overlay` (or a custom overlay token).
```typescript
// button.tsx line 14
"bg-destructive text-white hover:bg-destructive/90 ..."
// sheet.tsx line 39
"... bg-black/50"
```
- **Recommendation:** Override these in the shadcn component files to use semantic tokens (`text-destructive-foreground`, `bg-overlay`). This is a one-time fix.

**H4. All page-level components are client components**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/app/[locale]/(dashboard)/routes/page.tsx`, line 1
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/app/[locale]/(dashboard)/stops/page.tsx`, line 1
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/app/[locale]/(dashboard)/schedules/page.tsx`, line 1
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/app/[locale]/(dashboard)/documents/page.tsx`, line 1
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/app/[locale]/(dashboard)/chat/page.tsx`, line 1
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/app/[locale]/login/page.tsx`, line 1
- **Standard:** Architecture
- **Details:** Every page (except the Dashboard) uses `"use client"` at the top level. This means the entire page tree is client-rendered, losing the benefits of React Server Components: larger JavaScript bundles, no streaming, no server-side data fetching. The dashboard page correctly uses server components, proving the pattern works.
- **Recommendation:** Refactor page components to be server components that fetch initial data on the server, then pass it to client sub-components. Move `"use client"` to the interactive wrapper components only.

**M1. `listModels()` returns `Promise<unknown>` -- weak typing**


**M2. Mock dashboard metrics contain hardcoded English strings**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/lib/mock-dashboard-data.ts`, lines 5-31
- **Standard:** i18n
- **Details:** The `MOCK_METRICS` array contains hardcoded English titles like `"Active Vehicles"` and subtitles like `"Compared to 330 last month"`. While the dashboard page translates the `title` via `t()`, the `subtitle` and `value` fields in the mock data are raw English strings. The `subtitle` is properly overridden in the page (`t("metrics.comparedToLastMonth")`), but the `value` strings (`"342"`, `"94.2%"`, `"3"`, `"87%"`) are passed through directly. These numeric values are fine as-is, but the structure of the mock data is misleading -- it contains untranslated English that could leak if the page implementation changes.
- **Recommendation:** Use i18n key references in mock data (like the events do with `"eventTitles.busFleetInspection"`) or strip the English text and only keep the values needed.

**M3. Unauthorized redirect in middleware uses hardcoded `/unauthorized` path without locale**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/middleware.ts`, line 28
- **Standard:** Architecture / i18n
- **Details:** When a user lacks permission, the middleware redirects to `/unauthorized` without a locale prefix, while the login redirect correctly uses `/lv/login`. The unauthorized page lives at `/[locale]/unauthorized/`, so the redirect should include the locale.
```typescript
return NextResponse.redirect(new URL("/unauthorized", req.url));
// Should be:
return NextResponse.redirect(new URL(`/${locale}/unauthorized`, req.url));
```
- **Recommendation:** Extract the locale from the pathname and include it in the unauthorized redirect URL.

**M4. Unauthenticated redirect always goes to `/lv/login` regardless of current locale**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/middleware.ts`, line 18
- **Standard:** i18n
- **Details:** If an English-locale user (`/en/routes`) is not authenticated, they are redirected to `/lv/login` (Latvian). The redirect should preserve the user's current locale.
```typescript
return NextResponse.redirect(new URL("/lv/login", req.url));
```
- **Recommendation:** Extract the locale from the request pathname and redirect to `/${locale}/login`.

**M5. `useChatAgent` hook has stale closure over `messages`**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/hooks/use-chat-agent.ts`, lines 22-68
- **Standard:** Performance / Code Quality
- **Details:** The `sendMessage` callback depends on `messages` state in its dependency array, which means it is recreated on every message change. More critically, on line 38, `const allMessages = [...messages, userMessage]` captures the `messages` state at the time of the closure, but `messages` was just updated via `setMessages` on line 33 -- since setState is async, `allMessages` may not include the latest state. The `retryLast` function has the same pattern.
- **Recommendation:** Use a ref for messages or use the functional updater pattern to access current state.

**M8. `document.cookie` set without `SameSite` attribute**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/components/locale-toggle.tsx`, line 11
- **Standard:** Security
- **Details:** The locale cookie is set without `SameSite` attribute. Modern browsers default to `Lax`, but explicit declaration is a security best practice.
```typescript
document.cookie = `${name}=${value};path=/;max-age=${maxAge}`;
```
- **Recommendation:** Add `SameSite=Lax` to the cookie string.

**M9. Pagination does not handle large page counts properly**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/src/components/documents/document-table.tsx`, line 291
- **Standard:** Performance / UX
- **Details:** The document table pagination renders ALL page numbers: `Array.from({ length: totalPages })`. With thousands of documents, this could render hundreds of page buttons. The trip-table correctly limits to 5 pages (`Math.min(totalPages, 5)`), but the document table does not.
- **Recommendation:** Add pagination truncation (e.g., show first, last, and pages around current).

**M10. `chat/page.tsx` has no RBAC enforcement for viewer role**
- **File:** `/Users/Berzins/Desktop/VTV/cms/apps/web/middleware.ts`, lines 6-9
- **Standard:** Security
- **Details:** The `viewer` role's permissions list does not include `/chat`, which is correct middleware enforcement. However, the chat page link appears in the sidebar for all roles (need to verify). If the sidebar shows the chat link to viewers, they would see an unauthorized redirect, which is poor UX.
- **Recommendation:** Conditionally render the chat sidebar link based on the user's role.
